Enhancement 1: Software Design and Engineering

The artifact I selected is the Grazioso Salvare animal shelter dashboard I created for CS 340. It is a Python Dash application that displays shelter records in an interactive table and updates a chart and map based on user-selected rescue filters. The original version was developed during CS 340 as the final project and was designed around a MongoDB-backed dataset and a single-file, course-focused implementation. I included this artifact in my ePortfolio because it demonstrates applied software development skills that are easy to evaluate, including UI construction, data flow from a persistence layer, and business rules that connect user input to meaningful results. I selected it because it also had clear room for professional improvement. The enhanced version refactors the code into distinct layers so the Dash layout and callbacks are separated from query building, input validation, scoring/ranking logic, and database operations. I added logging and predictable error handling so the application behaves consistently when the database is unavailable, when filters return no results, or when invalid inputs are received. I also improved security practices by moving configuration out of the code into environment-based settings, validating filter inputs to prevent unsafe query patterns, and aligning the database layer with a least-privilege approach. Finally, I added automated tests for the non-UI components to improve reliability and make the project safer to maintain. These changes support the outcomes I planned to address in Module One by demonstrating professional-quality design decisions, clear separation of concerns, and technically sound implementation practices. They also strengthen the security mindset outcome through safer configuration and stricter input boundaries. I do not need major updates to my outcome-coverage plan, but this milestone is now clearly focused on software design and engineering, while later milestones can expand algorithmic depth and database-specific improvements. Enhancing the artifact reinforced the importance of separating responsibilities so the application is testable and maintainable. The biggest challenge was untangling callback-driven logic from database operations without changing user-facing behavior. Adding validation boundaries and consistent failure behavior also required careful handling so the dashboard remained usable while being more secure and predictable.

Enhancement 2: Algorithms and Data Structure

The artifact demonstrates proficiency across multiple software engineering disciplines: the RankingAlgorithm class showcases algorithmic thinking through a transparent, weighted scoring system with proper separation of concerns; the enhanced mongo_repo.py exemplifies database optimization by implementing server-side pagination and multi-field sorting; and the frozen dataclasses (MatchCriteria, BreedScore) reflect knowledge of immutable data structures for preventing bugs. The improvement from the original codebase is significant as the original had no ranking capability, no server-side pagination, and minimal test coverage. The enhancement added a complete scoring framework with 101 tests, reduced response time through database-level operations, and introduced proper data structures with comprehensive documentation. Each component is intentionally designed to be testable, maintainable, and scalable. I successfully met the course outcome planned in Module One regarding algorithms and data structures as the primary enhancement. The scoring algorithm directly addresses the requirement to add an enhancement that is clearly algorithmic rather than just filtering data since it implements a three-dimensional weighted scoring system (breed, age, sex) with transparent component breakdown, demonstrating both algorithmic design and appropriate data structure selection through frozen dataclasses. I also exceeded the initial scope by addressing a secondary outcome: performance optimization through server-side pagination, which was not explicitly planned but naturally emerged as a critical gap when evaluating the original architecture. My outcome-coverage plan is unchanged so far. Building this enhancement showed me that designing the data structures first made everything cleaner. The hardest part was testing edge cases because small off-by-one issues could slip through. Iterating on the scoring weights helped keep it explainable and writing 101 tests pushed me to handle failure cases like missing or invalid data.

Enhancement 3: Databases

I successfully met the course outcome planned in Module One regarding databases as the primary enhancement. Refactoring the dashboard into clearer layers, adding server-side pagination and sorting, implementing a ranking algorithm, and strengthening the database layer with validation, aggregations, and indexing all provide direct evidence of those goals. Enhancing the artifact reinforced how much maintainability and reliability depend on structure. Separating UI, service logic, and database access made the system easier to reason about and made testing possible without relying on the Dash layer. I also learned that performance improvements are often design choices. Moving sorting, pagination, and rollups into the database reduces UI work and scales better. The main challenges were untangling the original callback-driven code without changing user-facing behavior and ensuring the new validation and sanitization logic handled messy or incomplete records without breaking the dashboard components.
